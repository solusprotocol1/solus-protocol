Security Architecture

How S4 Ledger protects defense logistics data across all classification levels

Core Principle: Hash-Only Architecture

S4 Ledger never stores, transmits, or processes original record content on any public infrastructure. Only an irreversible 64-character SHA-256 hash is anchored to the XRP Ledger.

On-chain: A 64-character SHA-256 hash (e.g., a7ffc6f8bf1ed7...)
Never on-chain: Original records, names, NSNs, part numbers, locations, technical data, or any identifiable content

Why this matters: SHA-256 is a cryptographic one-way function. It is computationally impossible to reconstruct the original data from the hash. This means even if an adversary reads every transaction on the XRP Ledger, they gain zero knowledge about the underlying defense record.

Classification-Level Support

Because only the hash touches external infrastructure, S4 Ledger can support records at any classification level when deployed appropriately:

Unclassified CUI / FOUO Secret (SIPRNet) Top Secret (JWICS)

How It Works by Classification

  • Unclassified / CUI: Hash computation via S4 Ledger cloud API or on-premises SDK. Hash anchored to XRPL public mainnet. Original data stays in your systems.
  • Secret (SIPRNet): S4 Ledger deployed as an on-premises appliance within the SIPRNet boundary. Hash computed locally on the classified network. Only the unclassified hash is transmitted (via cross-domain solution or manual review) to a gateway that anchors to XRPL. The hash itself is not classified because it cannot be reversed.
  • Top Secret (JWICS): Same architecture as Secret, deployed within the JWICS enclave. Hash egress through approved cross-domain solution. All TS data remains within the TS boundary at all times.

Note: Classified deployments require accreditation by the Authorizing Official (AO) and a cross-domain solution approved by the Defense Information Systems Agency (DISA). S4 Ledger provides the technology; the customer provides the authorization.

Technical Security Controls

  • Encryption in Transit: TLS 1.3 enforced on all API and web connections
  • Encryption at Rest: AES-256 for any stored metadata
  • Hash Integrity: SHA-256 (FIPS 180-4 compliant) with HMAC verification
  • API Authentication: Per-customer API keys with role-based access control
  • Rate Limiting: Tier-based request quotas to prevent abuse
  • Audit Logging: All API operations logged with timestamps and user context
  • Zero Data Retention: Original record content is never persisted on S4 Ledger infrastructure
  • Immutable Audit Trail: Anchored hashes on XRPL cannot be altered, deleted, or tampered with
